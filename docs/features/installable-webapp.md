# Feature 7: Installable Web App (PWA)

**Status:** Proposed

## Summary

Make Merchant Scroll installable on smartphones and desktops as a Progressive Web App (PWA). Users can add the site to their home screen and open it in a standalone app window instead of the browser chrome — it looks and feels like a native app while remaining the same static site.

## Motivation

Merchant Scroll is already mobile-friendly (vertical carousel, swipe navigation, tap-to-preview cards), but it still lives inside the browser. Players who check decklists frequently must:

- Open the browser, type the URL or find a bookmark, and wait for the tab chrome to load
- Compete for attention with other tabs and browser UI
- Miss the quick-launch experience of a home-screen icon

An installable PWA removes this friction. A tap on the home screen opens Merchant Scroll full-screen, with the dark theme and carousel UX unchanged. No app store submission, no backend, and no change to the static GitHub Pages deployment model.

## Proposed Behavior

### Installation

1. On **Android (Chrome/Edge)**: the browser shows an "Install app" or "Add to Home screen" prompt when installability criteria are met; users can also install from the browser menu
2. On **iOS (Safari)**: users tap **Share → Add to Home Screen** (iOS does not show an automatic install banner for PWAs)
3. On **desktop (Chrome/Edge)**: an install icon appears in the address bar; the app opens in its own window without tabs or URL bar

### Installed Experience

- Opens in **standalone** display mode (no browser address bar or tab bar)
- Uses a dedicated **app icon** on the home screen / app launcher
- **Splash screen** on launch (generated by the OS from manifest name, background color, and icon)
- **Theme color** (`#0f0f0f`, already set) colors the status bar on mobile
- All existing features work unchanged: carousel navigation, card previews, deck export, player profiles

### Offline Behavior

The site is data-driven and updates every hour, so full offline browsing is not the primary goal. Minimum viable offline support:

- The app shell (HTML, CSS, JS) loads from cache when offline, showing a clear "You're offline — deck data unavailable" message instead of a broken page
- Fresh deck data still requires network access (same as today)

Optional later enhancement: cache the most recently viewed tournament JSON files for limited offline reading.

## Technical Approach

### Web App Manifest

Add `manifest.webmanifest` at the site root and link it from `index.html`:

```html
<link rel="manifest" href="/manifest.webmanifest" />
```

Example manifest:

```json
{
  "name": "Merchant Scroll",
  "short_name": "Merchant Scroll",
  "description": "Browse MTGO Pauper decklists in a vertical carousel",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#0f0f0f",
  "theme_color": "#0f0f0f",
  "orientation": "portrait-primary",
  "icons": [
    { "src": "/assets/icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/assets/icons/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any" },
    { "src": "/assets/icons/icon-512-maskable.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ]
}
```

### App Icons

Generate PNG icons from the existing brand assets (`assets/favicon.ico`, `assets/preview.jpg`):

| Size | Purpose |
|------|---------|
| 192×192 | Android home screen, install prompt |
| 512×512 | Android splash screen, Play Store–style listings |
| 512×512 maskable | Android adaptive icons (safe zone padding) |
| 180×180 | Apple touch icon (`<link rel="apple-touch-icon">`) |

Add to `index.html`:

```html
<link rel="apple-touch-icon" href="/assets/icons/apple-touch-icon.png" />
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
<meta name="apple-mobile-web-app-title" content="Merchant Scroll" />
```

### Service Worker

Add a minimal service worker (`sw.js`) registered from `index.html`:

1. **Install**: precache the app shell — `index.html`, `style.css`, `manifest.webmanifest`, and icon files
2. **Fetch**: network-first for `assets/pauper/**` JSON (deck data must stay fresh); cache-first for static shell assets
3. **Activate**: clean up old cache versions on deploy

Registration (in `index.html` or a small inline script):

```javascript
if ("serviceWorker" in navigator) {
  navigator.serviceWorker.register("/sw.js");
}
```

### GitHub Pages Considerations

- Manifest and service worker must be served from the **site root** (or same scope as `start_url`)
- Service worker scope is limited to its directory; placing `sw.js` at `/sw.js` covers the whole site
- HTTPS is already provided by GitHub Pages (required for service workers)
- Cache-bust the service worker on each deploy by changing a `CACHE_VERSION` constant inside `sw.js`

### No Backend Changes

The PWA layer is entirely frontend:

- No changes to the Python pipeline or GitHub Actions crawler
- No new dependencies in the build — vanilla JS service worker, static manifest file
- Icons can be generated once manually or via a one-off script

## Installability Checklist

Browsers require these criteria before showing an install prompt (Chrome on Android):

| Requirement | Status today | Action |
|-------------|--------------|--------|
| Served over HTTPS | Yes (GitHub Pages) | None |
| Valid web app manifest | No | Add `manifest.webmanifest` |
| Registered service worker | No | Add `sw.js` |
| Manifest has `name`, `short_name`, `start_url`, `display`, `icons` (192 + 512) | No | Fill manifest |
| User has engaged with the site (heuristic) | N/A | Automatic after criteria met |

## Dependencies

| Dependency | Required for |
|------------|-------------|
| Existing static frontend | App shell to cache |
| GitHub Pages HTTPS | Service worker and install prompt |
| Brand icon assets | Generating PWA icon set |

No dependency on other feature docs.

## Open Questions

- Should we show a custom in-app "Install Merchant Scroll" banner for eligible browsers, or rely only on the browser's native prompt?
- How aggressive should JSON caching be? Network-first keeps data fresh but means no offline deck browsing; stale-while-revalidate could show last-fetched decks when offline
- Is `orientation: portrait-primary` correct, or should landscape be allowed on tablets?
- Should the service worker skip caching Scryfall card image requests (they are already fetched client-side from `cards.scryfall.io`)?
- Do we need a `screenshots` array in the manifest for richer install UI on Android/desktop?
- iOS PWA limitations: no push notifications, limited storage, and Safari may evict cached data after ~7 days of non-use — is minimal offline support acceptable?
